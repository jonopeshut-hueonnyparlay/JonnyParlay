## CRIT Findings

### CRIT-1 (A1-023) — `_pick_log_lock()` yields even when lock cannot be acquired

**File:** `engine/run_picks.py` (approx. lines 49–51)  
**Impact:** Two concurrent processes can both hold the pick_log "lock" simultaneously, silently corrupting pick_log.csv with interleaved writes.

**Detail:**
The `_pick_log_lock()` context manager calls `filelock.FileLock.acquire(timeout=N)` but does not check the return value before yielding. When `FileLock.acquire()` times out on Windows it raises `filelock.Timeout`; the code catches this exception and then falls through to `yield` anyway. Any caller inside the `with _pick_log_lock():` block proceeds assuming exclusive access even though it has none.

The rowcount sidecar guard (T2c) runs inside this block and assumes exclusive access — if two processes write simultaneously the sidecar check itself races.

**Fix:** Re-raise `filelock.Timeout` (or any lock failure) before the `yield`. The caller should abort the write rather than proceeding without the lock.

### CRIT-2 (A1-013) — `auto_r12_from_log()` reads file after lock timeout

**File:** `engine/run_picks.py` (approx. lines 817–819)  
**Impact:** Same race window as CRIT-1. `auto_r12_from_log()` acquires the lock with a short timeout then proceeds to read/iterate pick_log.csv even if the acquisition failed. Stale or partially-written data is silently consumed.

**Detail:**
Pattern is:
```python
try:
    lock.acquire(timeout=2)
except filelock.Timeout:
    pass  # falls through to CSV read below
```
The `pass` means on lock failure the function reads the file unprotected. If the CLV daemon is mid-write to pick_log.csv at this moment, `auto_r12_from_log()` may read a truncated file, mis-count recent losers, and fire (or suppress) an R12 cooldown incorrectly.

**Fix:** Return early (empty result or cached value) on lock timeout; do not proceed with file read.

### CRIT-3 (G6-001) — `backtest_projections.py` manual CSV quote parser fragile

**File:** `engine/backtest_projections.py`  
**Impact:** Silently drops or miscounts picks in backtest results if any CSV value contains a double-quoted field. Backtest accuracy statistics are unreliable.

**Detail:**
The manual byte-level CSV parser uses:
```python
q = stripped.count(b'"') - stripped.count(b'""') * 2
```
This formula overcounts quote depth for escaped quotes inside fields (e.g., `"he said ""hi"""` is a valid CSV field). The parser can exit quoted mode mid-field or fail to exit at end-of-field, producing incorrect row splits. Using `csv.reader` with `StringIO` would handle all RFC-4180 quoting correctly.

**Fix:** Replace manual byte parser with `csv.reader(io.StringIO(line.decode('utf-8')))`.

### CRIT-4 (G6-007) — `backtest_projections.py` P1 mode omits DNP guard

**File:** `engine/backtest_projections.py`  
**Impact:** Players who did not play (actual_min == 0) are included in P1-mode adjusted MAE computation. Division by zero or inclusion of DNP rows inflates apparent model accuracy.

**Detail:**
Standard (P0) mode has `if actual_min == 0: continue` before computing adjusted projections. P1 mode (role-conditional minutes scalar) applies the minutes adjustment without first checking `actual_min == 0`. DNP players with adjusted projections and zero actuals contribute large errors that are excluded in P0, making P1 appear worse than it is, or vice versa.

**Fix:** Add `if actual_min == 0: continue` guard at the top of the P1 per-player loop, consistent with P0.

### CRIT-5 (H-CRIT-001) — `test_pick_log_atomic_write.py` uses 27-column header

**File:** `tests/test_pick_log_atomic_write.py`  
**Impact:** The atomic write test constructs test rows with 27 columns. The canonical schema is 28 columns (schema_version=3 added `legs` as column 28). The test passes silently while testing the wrong schema — real 28-column rows will not be validated by this test.

**Detail:**
`CANONICAL_HEADER` in pick_log_schema.py has 28 fields. The test file hard-codes a 27-field list (pre-`legs` schema). The test writes, reads back, and asserts success — but on 27-field data that doesn't match production rows. A genuine atomic write bug affecting 28-column rows would not be caught.

**Fix:** Update test header to match current CANONICAL_HEADER (28 columns, `legs` last).

### CRIT-6 (H-CRIT-002) — `test_pick_log_schema.py` does not verify `legs` column position

**File:** `tests/test_pick_log_schema.py`  
**Impact:** The schema test verifies column count but not that `legs` is the canonical 28th (last) column. Code that appends `legs` at position 27 (0-indexed) would pass the test while breaking any downstream tool that reads `legs` by column index.

**Detail:**
The test asserts `len(CANONICAL_HEADER) == 28` but does not assert `CANONICAL_HEADER[-1] == "legs"` or `CANONICAL_HEADER.index("legs") == 27`. The grader, CLV daemon, and analyze_picks.py all rely on `legs` being the final field.

**Fix:** Add `assert CANONICAL_HEADER[-1] == "legs"` and `assert CANONICAL_HEADER.index("legs") == 27`.

### H3 (A1-002) — Auto R12 cooldown window off-by-one

**File:** `engine/run_picks.py` (`auto_r12_from_log()` cooldown window, approx. line 820 region)  
**Impact:** A 5-day R12 cooldown window returns only 4 days of picks, potentially firing R12 one day early or failing to fire when 5 full days of losers have accumulated.

**Detail:**
The window uses `>= today - timedelta(days=5)` for inclusive lower bound, yielding rows for days D-4 through D+0 = 5 rows max. But the intent is to look back 5 full calendar days (D-5 through D-1), excluding today's not-yet-graded picks. The off-by-one error means on the 5th day of a losing streak, R12 does not trigger until the 6th.

**Fix:** Adjust the window boundary to `>= today - timedelta(days=6)` and `< today`, or document the current behavior as intentional.

### H8 (A1-026) — Premium VAKE sizing `max(final, 0.50)` inflates T3 picks

### H10 (A1-011) — `_load_cache()` catches ALL exceptions, including disk I/O errors

**File:** `engine/run_picks.py` (`_load_cache()`, Odds API cache layer)  
**Impact:** A disk I/O error (full disk, permissions error, corrupted cache file) is silently treated as "cache miss." The engine proceeds to make live API calls — but if the API is also down (the most likely scenario when cache was needed), the run silently fails to get odds and posts zero picks.

**Detail:**
```python
except Exception:
    return {}
```
This catch-all masks `OSError`, `PermissionError`, `IOError`. Distinguish between "file doesn't exist" (expected cache miss → return `{}`) and "file exists but unreadable" (operational error → log.warning + return `{}`).

**Fix:** Catch `FileNotFoundError` silently; re-raise or `log.warning()` for other `OSError` subclasses.

### H15 (B1-001) — REB Bayesian shrinkage applies to ALL thin samples, not just cold_start

**File:** `engine/nba_projector.py` (T6 REB shrinkage implementation)  
**Impact:** Players with 1–9 recent games (e.g., returning from injury, traded mid-season) have their rebound rate partially replaced by a positional prior even though they have real recent data. Cold_start players (0 games) are correctly shrunk; thin-sample non-cold_start players are incorrectly shrunk.

**Detail:**
T6 implemented REB Bayesian shrinkage intended for cold_start players. The implementation condition is `if _reb_n_games < threshold` rather than `if _reb_n_games == 0`. A player with 5 recent games who is averaging 8.0 RPG gets partially shrunk toward a 6.3 RPG positional prior (SF), producing a biased projection despite having meaningful real data.

**Fix:** Wrap the shrinkage block in `if _reb_n_games == 0:` (or `< MIN_GAMES_FOR_TIER`), matching the cold_start definition used elsewhere. For 1–9 game thin samples, use the observed rate without shrinkage (or use a much weaker prior weight).

### H22 (D1-003) — Discord recap duplicate-post risk on webhook retry failure

**File:** `engine/grade_picks.py` (Discord recap post guard)  
**Impact:** If the Discord webhook call fails on first attempt and is retried, the guard is marked only after the successful post. A race condition between the retry and a concurrent grader invocation (e.g., `--repost`) can post the recap twice to #daily-recap.

**Detail:**

### H23 (G1-003) — `weekly_recap.py` Discord guard fallback is non-atomic

**File:** `engine/weekly_recap.py` (guard fallback path, approx. line 580 region)  
**Impact:** If `discord_guard` module is unavailable (import error), the fallback uses a plain file read/check/write sequence. A TOCTOU race between two concurrent Sunday invocations causes both to see the guard as unset and both to post the weekly recap.

**Detail:**
The fallback pattern is:

This is not atomic. Two processes can both read before either writes. The `discord_guard` module itself is atomic (filelock-protected); the fallback loses this guarantee.

**Fix:** Remove the fallback entirely (require `discord_guard` module) or replicate its filelock pattern in the fallback.

## MED Findings

### M1 (A3-020) — Rowcount sidecar updated even if pick-log I/O fails mid-loop

**File:** `engine/run_picks.py` (`log_picks()`, rowcount sidecar update)  
**Impact:** If the CSV write loop is interrupted mid-way (e.g., disk full, exception on row 3 of 5), the sidecar is updated with `len(new_picks)` (5) even though only 3 rows were written. On the next run, the guard compares actual rowcount (3 written) against expected (5) and incorrectly aborts the write.

**Fix:** Only update the sidecar after the full write loop completes successfully. Use a try/finally to ensure the sidecar update is conditional on write success.

### M2 (A3-004) — `log_picks()` uses `adj_edge` with no fallback

**File:** `engine/run_picks.py` (`log_picks()`)  
**Impact:** If a pick dict lacks `adj_edge` (e.g., from a manual or legacy pick dict), `log_picks()` raises KeyError. Since `adj_edge` is set in sizing but some code paths that call `log_picks()` may bypass sizing, this is a latent crash.

**Fix:** Change to `pick.get("adj_edge", pick.get("edge", 0.0))` with a log.warning if `adj_edge` is missing.

### M3 (B1-021) — AST projection: training basis (team_pace) vs. projection basis (game_pace) mismatch

**File:** `engine/nba_projector.py`  
**Impact:** AST rate trained on `team_pace` (season average) is applied with `game_pace` (matchup-specific). For fast-vs-slow matchups, this introduces a ~3% directional bias in AST projections.

**Detail:**
The pace elasticity exponent was fitted against team_pace as the independent variable. When game_pace is substituted at projection time, the exponent is misapplied. This doesn't affect PTS (which uses per-game rather than pace-adjusted rates) but does affect AST and potentially BLK/STL.

**Fix:** Either retrain the pace elasticity against game_pace, or document that team_pace and game_pace are used interchangeably and quantify the bias in the backtest.

---

### M4 (B1-003) — `career_min_prior` returning 0.0 silently skips player

**File:** `engine/nba_projector.py` (cold_start minutes path)  
**Impact:** If `get_player_career_avg_minutes()` ever returns 0.0 (rather than None), the guard `if career_prior:` evaluates to False and the player is silently skipped (projected 0 minutes), producing a bogus projection.

**Detail:**
A player who genuinely averaged 0 minutes historically (e.g., two-way player, deep bench emergency call-up) would incorrectly trigger the None-guard path. The guard should be `if career_prior is not None:` rather than truthiness check.

**Fix:** Change `if career_prior:` to `if career_prior is not None:`.

### M5 (B1-012) — BLK and STL both use `_proj_poss_stl` for possession calculation

**File:** `engine/nba_projector.py`  
**Impact:** BLK and STL use the same possession elasticity function with the same exponent (0.30), which may be coincidentally correct but will silently apply the wrong exponent if STL's exponent is ever updated without updating BLK's.

**Fix:** Define separate `_proj_poss_blk()` function (even if initially identical to `_proj_poss_stl()`) to make the independence of the two stat types explicit in the code.

### M6 (B1-017) — REB decomposition uses prior_n=15, fallback uses prior_n=12

**File:** `engine/nba_projector.py` (REB decomposition and fallback paths)  
**Impact:** Minor inconsistency in Bayesian prior weight. Cold_start players who go through the decomposition path have their prior downweighted differently than those who hit the fallback path.

**Fix:** Standardize on one prior_n value for REB shrinkage. Choose 12 or 15 based on validation and apply consistently across both paths.

### M7 (B1-019) — Team 240-min constraint doesn't recompute player quantiles

**File:** `engine/nba_projector.py` (`constrain_team_totals()` post-scaling)  
**Impact:** After applying the team minutes scale factor, p25 and p75 distribution quantiles stored in the projection object are not updated. Downstream code that reads `proj["p25"]` or `proj["p75"]` for sizing confidence intervals uses pre-constraint quantiles.

**Fix:** After applying the scale factor to `proj["proj_min"]`, recompute `p25 = proj * 0.85` and `p75 = proj * 1.15` (or whatever the quantile formula is) on the scaled value.

### M8 (C3-003) — `compute_defensive_splits()` has no minimum-games filter

**File:** `engine/projections_db.py`  
**Impact:** A team that faced a given stat category only once (1-game sample) gets a defensive split ratio stored as authoritative. A 1-for-1 or 0-for-1 sample produces extreme outlier ratios (e.g., 3.2x or 0.0x) that bias matchup adjustments for that team.

**Fix:** Add `HAVING count(*) >= 5` (or configurable `MIN_SPLIT_GAMES`) to the defensive splits query, consistent with the N≥5 gate already used in `clv_report.py`.

### M9 (C3-006) — Implied total coverage warning issued after projections already computed

**File:** `engine/generate_projections.py`  
**Impact:** The warning "X% of games missing Vegas totals" is issued after `generate_all_projections()` has already run. The operator cannot act on this warning before projections are finalized.

**Fix:** Check implied total coverage before calling `generate_all_projections()`. If coverage is below threshold, issue the warning and optionally abort (with `--no-constraint` flag to proceed anyway).

### M11 (D1-006) — Monthly footer `(month % 12) + 1` edge case

### M12 (D1-007) — `grade_prop()` falls through to "under" for invalid direction

### M14 (F4-002) — `engine_logger.py` idempotency key collision on `log_path=None`

**File:** `engine/engine_logger.py`  
**Impact:** `log_path=None` and `log_path=""` produce the same idempotency key (`str(None)` and `str("")` are different, but if the key is constructed as `f"{name}{log_path}"`, then `name+"None"` vs `name+""` — the real risk is if `log_path` is hashed as `hash(None) == hash(None)`, which collides with itself across calls). On the second call with `log_path=None`, the file handler is silently skipped, meaning no log file is written for the second logger initialized with `None` path.

**Fix:** Use `repr(log_path)` as the idempotency key component instead of `str(log_path)`.

### M15 (F7-001) — `injury_parser.py` catches all exceptions, returns empty DataFrame

**File:** `engine/injury_parser.py`  
**Impact:** "API is down" and "no injuries today" both return an empty DataFrame. The engine cannot distinguish between a healthy parse with zero injuries and a failed parse. If the injury API is silently down, all players are projected without injury adjustment.

**Fix:** Catch specific exceptions (`requests.RequestException`, `KeyError`, `ValueError`) separately. For network errors, log.warning with the exception message. For "no injuries" (empty response), log.debug. Return a tagged result (e.g., a `(DataFrame, ok: bool)` tuple) so callers can decide whether to proceed.
