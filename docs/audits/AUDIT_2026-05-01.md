## HIGH Findings

### H1 — go.ps1 syncPairs loop overwrites root shims on every run (L16 regression)

**File:** `go.ps1` (section [3/8], `$syncPairs` loop)
**Impact:** Every invocation of `go.bat` / `go.ps1` copies 7 engine/ source files over their corresponding root shims, silently reverting L16's shim architecture on Jono's production machine.

**Detail:**
L16 (closed Apr 30 2026) converted root entry-point files to 5-line `runpy.run_module()` shims with the explicit goal of eliminating copy-sync drift. CLAUDE.md states: *"No sync step needed — root entry points are 5-line shims."*

However, go.ps1 still contains a `$syncPairs` array and a `Get-FileHash` comparison loop that runs on every invocation. Because root shims and engine source files have different content, the hash check ALWAYS fires, and `Copy-Item $eng $root -Force` overwrites the shim with the full engine file. On the next run, root files are back to being engine copies — the old pre-L16 model.

Files affected (exist as both root shim AND in syncPairs):
- `run_picks.py` <- `engine\run_picks.py`
- `grade_picks.py` <- `engine\grade_picks.py`
- `capture_clv.py` <- `engine\capture_clv.py`
- `analyze_picks.py` <- `engine\analyze_picks.py`
- `results_graphic.py` <- `engine\results_graphic.py`
- `weekly_recap.py` <- `engine\weekly_recap.py`
- `morning_preview.py` <- `engine\morning_preview.py`

Not affected (root shims exist but NOT in syncPairs): `clv_report.py`, `sgp_builder.py`.
Not affected (no root copy exists): `pick_log_schema.py`, `name_utils.py`.

**Consequence:** On Jono's production Windows box, root `run_picks.py` etc. are engine copies (thousands of lines), not shims. The engine works correctly either way. What is lost: the L16 guarantee of no-drift. L16's stated intent — that root files are permanent minimal shims requiring no maintenance — is not honored on Windows.

**Fix options:**
- A. Remove the `$syncPairs` loop from go.ps1 entirely (matches L16 intent: shims never drift by design).
- B. Update the loop to SKIP files where the root matches the shim pattern.
- C. Keep the loop, document that Windows production uses engine copies not shims, update CLAUDE.md.

Option A is cleanest and matches the stated L16 architecture.

### H2 — test_tail_guard.py broken by L16 shim conversion

**File:** `tests/test_tail_guard.py`
**Impact:** Three test assertions fail on every run post-L16 in Cowork/CI environments, producing a permanently broken test suite for critical production files.

**Detail:**
`test_tail_guard.py` guards CRITICAL_FILES by checking both `engine/<name>` AND root copies for:
1. The canonical tail: `if __name__ == "__main__":\n    main()\n`
2. Minimum size: `MIN_BYTES = 1_000`
3. Hash equality between `engine/` and root

L16 (Apr 30 2026) made root files 5-line shims. Root shims:
- End with `_runpy.run_module(...)` — NOT the canonical tail
- Are ~150 bytes — well below the 1000-byte floor
- Have different hashes than engine/ files

All three assertions fail for every file in CRITICAL_FILES that has a root copy. The test was written before L16 and never updated.

**Note:** Due to H1, on Jono's production machine root files get overwritten to engine copies by go.ps1, so the tests may pass there. In Cowork/CI environments (where go.ps1 hasn't run) they fail definitively.

**Fix:** Update `test_tail_guard.py` to:
- For `engine/<name>`: keep existing checks (canonical tail + >=1000 bytes)
- For root `<name>`: verify it IS a valid shim (<=20 lines, contains `runpy.run_module`, delegates to the correct module name)
- Remove hash-equality assertion (shims and engine files are intentionally different by design)

## MED Findings

### M1 — sgp_builder.py: P16 Negative Binomial not ported for 3PM

**File:** `engine/sgp_builder.py`
**Impact:** SGP win-probability calculations for 3PM props use a Normal distribution instead of the Negative Binomial model (P16) implemented in `run_picks.py`. Over-predicts tail probabilities for high-variance low-mean events (3PM), creating systematic SGP miscalibration for 3PM legs.

**Detail:** `run_picks.py` implements P16: `NB_STATS = {"3PM", ...}` with `NB_R` shape parameters, using `scipy.stats.nbinom.sf()` for win probability. `sgp_builder.py` has no equivalent — it uses a generic `_normal_win_prob()` for all stat types including 3PM. The Normal approximation significantly underestimates the probability of extreme values for 3PM.

**Fix:** Port `NB_STATS`, `NB_R`, and the NB win-prob branch from `run_picks.py` into `sgp_builder.py`'s win-prob calculation path.

### M2 — csv_writer.py: "S" position fallback for sixth_man is undocumented

**File:** `engine/csv_writer.py`
**Impact:** Players with `sixth_man` role who lack a DB position entry get "S" written to the `position` column. Downstream consumers (injury_parser, projections_db rotation queries) filter on "G"/"F"/"C" — "S" matches none, silently excluding sixth-man players from minutes-redistribution calculations when a starter is OUT.

**Detail:** The position-mapping fallback chain produces "S" for `role == "sixth_man"` with no DB position. `_normalise_position()` in injury_parser.py maps unknown values to "F" as a default, so the effective result is that sixth-men become forwards — correct directionally but implicit and untested.

**Fix:** Either (a) map `sixth_man` to "F" explicitly in csv_writer instead of "S", or (b) document "S" as a valid sentinel and add explicit handling in `_normalise_position()`.

---

### M3 — projections_db.py: get_team_avg_fga() lacks season_type filter

**File:** `engine/projections_db.py`, function `get_team_avg_fga()`
**Impact:** Team FGA averages used for 3PM context include regular-season AND playoff games from all matching seasons. Playoff pace and shot-selection differ significantly from regular season; cross-season-type data dilutes current-season signals.

**Detail:** The function queries with `season = ?` but no `game_type` / `season_type` predicate. If the DB stores playoff rows with the same season tag, they contaminate the regular-season average. The function also lacks an `ORDER BY game_date DESC LIMIT ?` clause, so stale early-season games contribute equally to recent games.

**Fix:** Add `AND game_type = 'regular'` (or equivalent) and `ORDER BY game_date DESC LIMIT ?` matching the pattern used elsewhere in `projections_db.py`.

### M4 — test_section23_observability.py: go.ps1 sync-pair tests validate outdated copy model

**File:** `tests/test_section23_observability.py` (`test_go_ps1_contains_all_sync_pairs`, `test_go_ps1_sync_pairs_point_to_engine_subdir`)
**Impact:** These tests assert that go.ps1 contains `$syncPairs` entries. Post-L16, the sync pairs are actively harmful (they overwrite shims — see H1), but these tests PASS and report the harmful behavior as correct. A fix for H1 (removing syncPairs) would break these tests, discouraging the correct repair.

**Fix:** After fixing H1, update these tests to verify the correct post-L16 behavior: either (a) syncPairs is absent from go.ps1, or (b) go.ps1 correctly handles shim files without overwriting them.
