## CRITICAL (14)

### C1 — nba_projector.py: compute_stl_blk_rates() called with team_pace instead of game_pace
**File:** `engine/nba_projector.py` ~line 1239  
**Impact:** STL and BLK projections systematically biased by ±1–3% in all games where team_pace ≠ opponent_pace. Analogous to M3 bug (AST rate using team_pace) — fixed for AST in session 3 but STL/BLK missed.  
**Fix:** Change `compute_stl_blk_rates(player_id, player_pos, team_id, opp_team_id, team_pace, ...)` to use `game_pace = (team_pace + opp_pace) / 2`.

### C2 — nba_projector.py: compute_tov_rate() called with team_pace instead of game_pace
**File:** `engine/nba_projector.py` ~line 1247  
**Impact:** TOV projections systematically biased. Same root cause as C1.  
**Fix:** Change `compute_tov_rate(player_id, team_id, team_pace, ...)` to use `game_pace`.

### C3 — nba_projector.py: playoff pace scaling lost when implied_total overrides _base_pf
**File:** `engine/nba_projector.py` ~line 1064–1068  
**Impact:** When `implied_total` is available from the Odds API, `_base_pf` is set directly from `implied_total * 2 / possessions` without applying the playoff pace scalar (`LEAGUE_AVG_PACE_PO / LEAGUE_AVG_PACE`). All playoff games with Odds API coverage get ~3% too-high possession count, inflating all counting stats.  
**Fix:** Apply playoff pace scalar to `game_pace` before computing possessions when `implied_total` path is taken.

### C4 — projections_db.py: get_player_last_appearance_days() missing try/except on date.fromisoformat()
**File:** `engine/projections_db.py` (R7 new function)  
**Impact:** Any malformed date string in the DB (e.g., partial game_date from a failed insert) causes an unhandled `ValueError`, crashing `project_player()` for cold_start players.  
**Fix:** Wrap `date.fromisoformat()` in `try/except ValueError`, log warning, return `None`.

### C5 — projections_db.py: upsert_projection() ON CONFLICT DO UPDATE missing columns
**File:** `engine/projections_db.py` (upsert_projection function)  
**Impact:** On re-run for the same player+game, `player_name`, `team_id`, `opp_team_id`, `is_home`, and all new projection columns are NOT updated — stale data persists in the DB. The `ON CONFLICT DO UPDATE SET` clause is incomplete.  
**Fix:** Add all columns to the SET clause in the ON CONFLICT path.

### C6 — capture_clv.py: implied_prob() unsafe for odds=0, NaN, inf
**File:** `engine/capture_clv.py` (implied_prob function)  
**Impact:** If bookmaker API returns `odds=0` (API error), `implied_prob()` computes `100 / (100 + 0) = 1.0` or divides by zero. NaN/inf odds write `"nan"` or `"inf"` strings directly to pick_log CSV, corrupting the ledger.  
**Fix:** Guard: `if not odds or not math.isfinite(odds): return None`. Callers must handle `None`.

### C7 — grade_picks.py: H23 non-atomic _save_guard fallback still present
**File:** `engine/grade_picks.py` ~lines 1764–1770  
**Impact:** H23 (session 3) removed the non-atomic fallback write from `weekly_recap.py` and `morning_preview.py` but missed the same pattern in `grade_picks.py`. On exception, the guard file is still written via direct `open()/json.dump()`, which can truncate the file if disk is full mid-write.  
**Fix:** Remove direct `open()` write on exception path; log warning only (same pattern as H23 fix in other files).

### C8 — weekly_recap.py: filter_week() excludes VOID picks
**File:** `engine/weekly_recap.py` ~lines 237–243  
**Impact:** `filter_week()` filters to `result in ("W", "L", "P")` — VOID picks (DNP/scratched player, functionally a push/refund) are excluded from the weekly win/loss/push totals. Weekly recaps under-count total bets placed and over-report win rate if any picks were voided.  
**Fix:** Add `"VOID"` to the allowed set, treating it identically to `"P"` (push) for record-keeping.

### C9 — run_picks.py: undefined `log` variable at ~line 1322
**File:** `engine/run_picks.py` ~line 1322  
**Impact:** In the odds cache corruption handler, `log.warning(...)` is used but `log` is not defined — the module uses `logger`. This code path crashes with `NameError: name 'log' is not defined` whenever the odds cache file is malformed, which then masks the actual cache corruption error.  
**Fix:** Replace `log.warning` with `logger.warning`.

### C10 — run_picks.py: _pick_log_lock() timeout path still yields (CRIT-1 regression check)
**File:** `engine/run_picks.py` ~lines 50–52  
**Impact:** CRIT-1 from session 1 required raising `_FileLockTimeout` on lock timeout. The fix message says it was applied, but the warning text "writing anyway (race risk)" suggests the yield path may still be present in some code paths. Requires code verification — if confirmed present, it's a CRIT regression.  
**Status:** **Verify before fix** — may be a false positive from the audit if CRIT-1 was fully applied.

### C11 — test coverage: no tests for get_player_career_game_count()
**File:** `tests/` (missing file)  
**Impact:** R7 added `get_player_career_game_count()` — zero test coverage. A bug in this function silently misclassifies cold_start sub-types.

### C12 — test coverage: no tests for get_player_last_appearance_days()
**File:** `tests/` (missing file)  
**Impact:** R7 added `get_player_last_appearance_days()` — zero test coverage. Date arithmetic bugs (off-by-one, timezone issues) go undetected.

### C13 — test coverage: no tests for cold_start sub-type classification
**File:** `tests/` (missing file)  
**Impact:** taxi/returner/new_acquisition branching logic in `project_player()` — zero coverage. Misclassification silently gives wrong minute caps.

### C14 — test coverage: no tests for CUSTOM_SHADOW_LOG / ENABLE_CUSTOM_CLV
**File:** `tests/` (missing file)  
**Impact:** CLV shadow fix (session 6) added CUSTOM_SHADOW_LOG wiring — zero tests. If the daemon's log_paths construction regresses, custom shadow CLV is silently never captured.

## HIGH (17)

### H1 — projections_db.py: get_team_pace() fallback still returns 99.5 not 100.22
**File:** `engine/projections_db.py` (get_team_pace function)  
**Impact:** R5 updated `LEAGUE_AVG_PACE = 100.22` in `nba_projector.py` but missed the `return 99.5` fallback in `get_team_pace()`. Any player/game with no pace data in the DB uses the stale fallback, affecting ~2–5% of projections (expansion players, trade deadline acquisitions).

### H2 — projections_db.py: get_player_trade_context() connection not closed in finally
**File:** `engine/projections_db.py` (get_player_trade_context function)  
**Impact:** DB connection leak if function raises mid-execution. Under load (many players), can exhaust SQLite connections.  
**Fix:** Wrap in try/finally, call `conn.close()` in finally.

### H3 — capture_clv.py: is_home logic uses OR instead of AND
**File:** `engine/capture_clv.py` ~line 681  
**Impact:** `if player_team == home_team or player_team == away_team` — both conditions can't be true simultaneously (a team can't be home AND away), but the OR means the wrong branch is taken for `is_home` when player_team matches neither team (unknown team). Should be `if player_team == home_team` (positive match) else `if player_team == away_team` (negative match) else unknown.

### H4 — capture_clv.py: SPREAD fallback conflates empty is_home with False (away)
**File:** `engine/capture_clv.py` (SPREAD CLV path)  
**Impact:** Legacy pick_log rows (before H4 fix Apr 2026) may have `is_home=""`. The SPREAD CLV path uses `is_home` to select the right team's spread odds. Empty string evaluates as falsy, so blank is_home is treated as "away" — corrupts CLV for pre-fix spread picks in any backfill run.

### H5 — run_picks.py: ReadTimeout not excluded from webhook retry
**File:** `engine/run_picks.py` (_webhook_post function)  
**Impact:** H22 (session 3) added split timeout `(5, 10)` but the audit found `ReadTimeout` (response timeout after body sent) is still being retried. A retry after ReadTimeout can cause duplicate Discord posts. H22 was supposed to exclude ReadTimeout from retries.  
**Status:** Verify if H22 fully applied the ReadTimeout exclusion or only added the split timeout.

### H6 — run_picks.py: KILLSHOT odds int() crashes on float odds string
**File:** `engine/run_picks.py` (KILLSHOT block)  
**Impact:** `int(pick.get("odds", 0))` crashes with `ValueError` if odds is `"-115.5"` (float string, valid American odds format). KILLSHOT post fails silently, pick never posted.  
**Fix:** `int(float(pick.get("odds", 0)))` or cast via `round(float(...))`.

### H7 — run_picks.py: bonus pick dict missing explicit `'legs': ""`
**File:** `engine/run_picks.py` (bonus pick row construction)  
**Impact:** Schema requires `legs` column in every row. Bonus dict missing `'legs': ""` causes the grader's JSON-first legs parse to fail, falling back to game-string parse which may not be present either.

### H8 — sabersim_backtest.py: persist=False causes N² regeneration
**File:** `engine/sabersim_backtest.py` (~line 90)  
**Impact:** `run_projections(persist=False)` is called inside the backtest loop — projections are recomputed from scratch on every iteration instead of reading from the DB. For a 30-game backtest, this is 30× slower than necessary.  
**Fix:** `persist=True` so projections are written to DB on first call; subsequent iterations read from cache.

### H9 — generate_projections.py: DB connection in coverage warning has no finally
**File:** `engine/generate_projections.py` (implied-total coverage warning block)  
**Impact:** If the warning query throws, connection leaks. Low-impact (runs once per day) but consistent with project's close-in-finally standard.

### H10 — generate_projections.py: --shadow + --run-picks silently shadows
**File:** `engine/generate_projections.py` (argument parsing)  
**Impact:** Passing both `--shadow` and `--run-picks` causes `--shadow` to win without warning the user that `--run-picks` was ignored. Should log a `WARNING: --run-picks ignored when --shadow is set`.

### H11 — nba_projector.py: stale "99.5 fallback" comment in pace block
**File:** `engine/nba_projector.py` (near LEAGUE_AVG_PACE usage)  
**Impact:** After R5 update to 100.22, inline comment still references "99.5 fallback". Misleading for future debugging. Minor but tracked at HIGH due to potential future confusion during calibration.

### H12 — clv_report.py: CLV sort magic number -99 vs -float("inf")
**File:** `engine/clv_report.py` (sort key)  
**Impact:** Sort key uses `-99` as sentinel for missing CLV. Any pick with actual CLV < -99 (extremely bad line, e.g., closing at +3000 after betting +100) would sort incorrectly. Should be `-float("inf")`.

### H13 — test_pick_log_atomic_write.py: doesn't assert `legs` at position 27
**File:** `tests/test_pick_log_atomic_write.py`  
**Impact:** CRIT-5/CRIT-6 from session 1 fixed the 28-col schema, but the atomic write test doesn't assert the `legs` column is the last (position 27). A schema regression could go undetected.

### H14 — test_daily_lay_builder_v2.py: per-leg gates test doesn't verify call count
**File:** `tests/test_daily_lay_builder_v2.py`  
**Impact:** The per-leg gates test only checks final output — doesn't assert that picks below `edge<0.025` or `cover_prob<0.58` are rejected at the per-leg level (vs. failing the whole parlay). A regression in gate ordering would not be caught.

### H15 — capture_clv.py: ghost-game check doesn't include pick_log_custom.csv consistently
**File:** `engine/capture_clv.py` (ghost-game checkpoint block)  
**Impact:** The CLV shadow fix adds `CUSTOM_SHADOW_LOG` to `log_paths` for CLV writing, but the ghost-game integrity check at startup may not check `pick_log_custom.csv` for checkpoint corruption. If the custom log gets a ghost game, the daemon proceeds without detecting it.  
**Status:** Verify the ghost-game block consistently includes pick_log_custom.csv.

### H16 — injury_parser.py: minute overrides not validated against active roster
**File:** `engine/injury_parser.py`  
**Impact:** Manual minute overrides in the injury file are applied to any player_name match — no check that the player is on the active roster for that game. A stale override from a previous game remains silently applied.

### H17 — backtest_projections.py: on_bad_lines="error" may be set too strictly
**File:** `engine/backtest_projections.py` (H25 from session 3)  
**Impact:** H25 set `on_bad_lines="error"` after pre-validating legacy rows. But if any legacy row has a column count != 27 AND != 28, pandas raises immediately instead of padding/skipping. The pre-validation pads 27→28 but any other count goes unhandled.
